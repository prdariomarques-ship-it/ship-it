# AI Operator Center — Phase 1

The first thing the user sees on `/admin`: an operational command center that answers **"what should I do right now?"** within five seconds of opening the page — not a passive list of facts.

## Deliberately not an LLM call

Every insight is a **deterministic rule over data the dashboard already fetches** (Goals, Tasks, Calendar, Jobs, CurrentContext) — not a prompt to a language model. Two reasons, both from the brief itself:

- **"No black-box AI. Every recommendation must explain why."** A rule like *"this task is overdue because `due_date < now`"* is inspectable, testable, and exactly reproducible. An LLM's reasoning for the same recommendation would not be, and would cost a real API call on every dashboard poll (5s/30s intervals) for no accuracy gain over a plain comparison.
- **Infrastructure frozen.** `frontend/lib/operator.ts` is a pure function, `buildOperatorInsights(input) → OperatorInsight[]` — no new backend endpoint, no new call to an LLM provider, unit-tested exactly like `goals/scoring.py` is on the backend.

## "What should I do right now?" — answered directly

`topInsight(insights)` picks the single most important actionable item — the first `urgent`-bucket insight, falling back to the first `today`-bucket one — and the UI renders it prominently, above everything else, under the heading **"Faça isso agora"** ("Do this now"). If nothing is actionable, the page says so plainly ("Nada urgente agora") instead of showing an empty-looking screen. This is the direct, literal answer to the five-second question, tested in `topInsight — 'what should I do right now'"` (`tests/operator.test.ts`).

## The four buckets

| Bucket | Meaning | Example |
| --- | --- | --- |
| 🔥 Requer atenção imediata (`urgent`) | Broken or overdue *right now* | A failed job, an overdue task, WhatsApp disconnected, a goal due tomorrow at <50% |
| ⚠️ Precisa de atenção hoje (`today`) | Needs a decision or action today, not yet broken | A goal awaiting approval, a task due within 48h, a calendar conflict, a goal due in 2-3 days at low progress |
| 💡 Oportunidades (`opportunity`) | Nothing wrong — a chance to get ahead | A goal ≥80% done, or "everything's quiet, good time for deep work" |
| 🤖 Automações disponíveis (`automation`) | Already running without you | A recurring goal, background jobs already in flight |

`recent_change` (count deltas between CurrentContext polls) is generated by the same function but **deliberately excluded from these four buckets** — it's history, not a to-do, and belongs to the Timeline (Phase 2), not the command center.

## Every insight, in full

Per the brief's exact requirement, every single insight carries:

| Field | What it is | How it's kept honest |
| --- | --- | --- |
| **Why this recommendation exists** | `reason` — the literal condition that fired | Always references real field values (a date, a count, a status) |
| **Confidence (0-100%)** | `confidence` — three fixed tiers: 95 (high/direct fact), 65 (medium/judgment call), 35 (low) | See "Why three fixed numbers" below — never a fabricated continuous score |
| **Expected impact** | `impact` — one sentence on what happens if you act (or don't) | Plain language, specific to the insight, not a generic phrase |
| **Estimated time required** | `estimatedMinutes` | A real, defensible number only for fixed actions (approve ≈1min, retry ≈1min, reschedule a conflict ≈5min, reconnect WhatsApp ≈10min, matching this session's own real QR-rescan experience); **`null` → "tempo variável"** for open-ended work (finishing a goal, doing an overdue task) — never a fabricated guess at someone else's effort |
| **One-click action** | `action` (approve goal / retry job) when a real backend action exists | Same mutations the dashboard MVP already wired (`POST /goals/{id}/approve`, `POST /admin/jobs/{id}/retry`) |
| **Dismiss / Snooze / Mark as completed** | Always available, on every insight | See below |

### Why three fixed numbers, not a spectrum

A continuous score like "83% confidence" would imply a model computed it — false precision for a rule-based system. 95/65/35 map to three real, distinct situations: **95** = a directly observed fact (a job's `status` is literally `failed`; a date comparison is literally true) — nothing to be uncertain about, so the confidence isn't hedging the fact, it's signaling "this isn't a guess." **65** = a judgment call the code makes explicit (a goal is "at risk" because of a threshold `*this system* chose*, like `<50% progress` and `≤3 days`) — reasonable, but a different threshold would flag differently, so it's marked as such. **35** exists in the type system for future rules that are more speculative than anything shipped today; nothing currently in `buildOperatorInsights` uses it — noted here so the tier isn't mistaken for dead code.

## Dismiss, Snooze, Mark as completed

Implemented in `frontend/hooks/use-operator-state.ts` — per-browser preference state in `localStorage`, **not a new backend endpoint**. This is a deliberate architecture choice, not a shortcut:

- Insight IDs are stable per underlying record (`missed-42` for task id 42), so dismissing/snoozing/completing one persists across polls for as long as the underlying condition keeps firing.
- The moment the *underlying* condition stops being true (the task is done, the job succeeded), that insight ID simply stops being generated — the dismissal state becomes irrelevant on its own, no cleanup needed.
- **Snooze** is a fixed 4-hour window (one click, no picker) — matching the dashboard's own explicit "real-time = short-interval polling" philosophy (`docs/DASHBOARD.md`) rather than adding a scheduling feature.
- This is a real, durable *preference* about what to show, not a business record — the exact same reasoning `observation/engine.py`'s in-memory cache already uses on the backend ("nothing in it is a new source of truth," see `docs/OBSERVATION_ENGINE.md`). A cleared browser/incognito session just sees everything again, which is correct: nothing was actually resolved, only hidden from view.

Tested in `frontend/tests/use-operator-state.test.ts` (7 tests: hide/dismiss/complete/snooze/restore, snooze expiry via fake timers, persistence across remounts, and that hiding one insight never affects another).

## Architecture

```
frontend/lib/operator.ts                buildOperatorInsights(), confidenceSummary(),
                                          topInsight() — pure functions, no React, no fetch
frontend/hooks/use-previous.ts            usePrevious<T>() — diffs CurrentContext polls
frontend/hooks/use-operator-state.ts       dismiss/snooze/complete, localStorage-backed
frontend/components/admin/
  AIOperatorCenter.tsx                    "Faça isso agora" + four bucket groups, each
                                          insight row showing why/impact/confidence/time/
                                          action + dismiss/snooze/complete
app/admin/page.tsx                        wired as the very first section (above Status
                                          dos sistemas); insights computed via useMemo
                                          from data already fetched by existing hooks
```

**No new backend endpoint.** Same principle the dashboard MVP established: every input to `buildOperatorInsights` was already being fetched.

## What this replaces

The former `SuggestedActionsPanel` (dashboard MVP) is deleted — its goal-approval/failed-job/overdue-task/next-goal coverage is now a subset of the Operator Center's eight rule categories, folded into the four buckets above, with the same action buttons preserved. One command center, not two overlapping panels.

## Known limitations

- **"What can be automated?" is grounded, not predictive** — it reports goals/jobs that are *already* automated rather than inferring new automation candidates from behavioral patterns (no historical-pattern data exists yet; see Phase 2, Memory & Timeline, for where that would start accumulating).
- **"Recently observed changes" only diffs counts, not content** and isn't shown in this command-center view at all — it's `Timeline`'s job (Phase 2).
- **Dismiss/snooze/complete are per-browser, not synced** — a different device or a cleared browser sees the full list again. Acceptable for a single-owner system (Dario OS is not multi-tenant); would need a real backend record if that ever changes.
- **Opportunity detection is intentionally conservative** (two rules) rather than exhaustive — grow with real usage feedback rather than guessing upfront.
